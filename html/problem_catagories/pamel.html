Introduction to Problem Classes of Cloud Computing

With the cloud computing field becoming multidisciplinary amongst other areas of STEM, we need to devise specialised frameworks to analyse various problems applicable to different styles of data. MapReduce paradigms focus on aggregating and reducing large datasets into a smaller results, TensorFlow models continuously adapt through deep-learning to provide better classification and inference utilities in software applications and Spark frameworks are the solution to datasets requiring iterative map/reduce algorithms that MapReduce otherwise cannot account the complexity for. I present below the classes of problems the frameworks in discussion specialise in solving.



MapReduce

Why MapReduce came about

Many implementations of special-purpose computation algorithms have been designed previously to process immense datasets such as (in the case of Google) “crawled documents, web request logs, etc.” in order to derive data such as “inverted indices, various representations of the graph structure of web documents, summaries of the number of pages crawled per host, the set of most frequent queries in a given day, etc.” (Dean et al., 2004).1 For these computations to be carried out within a short time, the input data must be distributed across hundreds or thousands of machines in a process called parallel computation.1 This triggered the need for a new cloud computing framework: MapReduce.

The initial conditions for a map/reduce framework to be useful is that 2:

  • The algorithm to be run on the dataset can be composed (such that you can combine results of the algorithm mapped on subsets together – aggregation).
  • The result produced is smaller than the initial set of data.
  • Calculation throughout mapping the algorithm on the data set does not depend on external data (only data provided by user is required).
  • The dataset is large enough that splitting it into subsets to distribute for independent computations will not weaken computation performance.

As a result of these conditions, the MapReduce framework came about. It is used for parallel batch processing of data-intensive problems that require arbitrarily dividing the workload into smaller workloads of equal/close size so that they can be distributed across virtual machines. This is so that the algorithm can work on the subsets and then combine the results of those computations to give a final result.3 This parallelisation of computation allows users to achieve fast performance on large-scale clusters on commodity hardware, and allows programmers exploit the large availability of resources of the cloud easily.1

How the framework is utilised

The motivation behind MapReduce allows it to be applied to many different fields such as “data clustering, machine learning, and computer vision” where the map/reduce algorithm is prevalent. This is so that the framework can deal with the parallelism. The mapping of the reduction algorithm will complete the iterations to finalise a result (Ekanayake et al., 2010).4

Google in particular have shown enthusiasm for the programming model as the model is straightforward to incoming programmers as it hides most of the work done (parallelisation, fault-tolerance, locality optimisation and load balancing) and its simplicity on being applied to many problems that can be expressed in terms of the conditions required.1 MapReduce (due to its large scalability and extensive fault tolerance) is also utilised by Google to handle data mining, machine learning and “large-scale web search applications” to Google Search (such as extracting specific data from a large dataset of indexed web pages) because the model has economically benefitted the company in terms of efficient use of data center resources.1, 5, 6

Limitations of MapReduce

MapReduce, as we have discussed, is beneficial for when we want to perform computations on large datasets in parallel, however, there are certain instances in which this framework is disadvantageous.

Since the framework is built to deal with acyclic data flow, MapReduce cannot be applied to recursive problems. A fundamental yet simple example can be calculating the next term in the Fibonacci sequence. MapReduce cannot be applied in this case as the computation of the current value is dependent on the computation of the two previous values i.e. f(k + 2) = f(k + 1) + f(k).7 Calculating this requires generating sub-problems which are dependent on the results of upcoming sub-problems. But MapReduce doesn't involve this kind of calculation.2

Another case in which MapReduce is not efficient is when the dataset in question is small enough that dividing the workload is not even required to efficiently compute the result.7 If the dataset is small enough to fit on a single machine, performing a single reduce(map(data)) operation will be more efficient rather than going through the whole MapReduce process requiring for the data to be partitioned.2 Taking all of this into consideration, we see how, as I stated above, MapReduce is used to solve problem which involve processing large data sets, that can be split up into distinct sub problems and processed concurrently.



TensorFlow

Why TensorFlow came about

TensorFlow is an open-source distributed processing framework developed by Google, primarily for use in deep-learning applications. It is a large-scale framework that includes a set of software libraries used to express and execute machine-learning algorithms on complex datasets.9, Since becoming open-sourced, TensorFlow allows researchers and developers to collaborate using this framework and the Google Cloud platform so new ideas can be tested and if these ideas succeeded, they could then be moved into products without having to rewrite the code.10 It is designed to function on any of the processors on which it is run. The TensorFlow model goes hand-in-hand with Google’s custom ASIC (application-specific integrated circuit) TPUs (Tensor Processing Units) that can be accessed through Google's Cloud platform. Since the framework and units are optimised for each other, users are able to write code in TensorFlow and use the platform to quickly process large data sets using machine-learning.8

How the framework is utilised

The framework is well-suited for solving problems in relation to classification and inference. Therefore Google has incorporated it in applications such as:

  • Text and voice search: Differentiating multiple voices using speech recognition and speech synthesis, filtering speech from background noise and learn to mimic voice patterns to convey natural sounding speech from text.8
  • Language translation: Handles differing sentence structures in all the languages Google Translate supports to improve translations.8 Deep-learning is also used to provide a visual translation system that allows you to take an image of text you want to translate using the camera. The framework then does the work to translate it but also render the translation visually onto the image.12
  • Image/video search: Classifying "objects, landmarks, people, sentiments, or activities" to improve image/video search quality.8
  • Rankbrain: Google-developed engine used in ranking results from Google Search.8
  • Smart Reply: Utility provided in Google's email app that creates automatic responses to emails in your inbox by reading the email's content then generating short phrase suggestions to reply with that you can choose from.11 The more the feature is used, the better adapted the utility becomes.

Limitations of TensorFlow

Although TensorFlow seems to be the preferred alternative to DistBelief for Google, TensorFlow still has its own shortcomings.

As TensorFlow is designed primarily for the TPU ASIC unit, during initialisation of the framework on a different machine, it will attempt to allocate itself to all GPU memory available, which in a cloud environment may be an interference as it will be unpredictable to know if the model will interact with other code accessing the same hardware in data centers and therefore interrupt any onging processes.13

Another issue at hand in relation to memory usage is that in order to maximise the performance of TensorFlow models, the number of GPUs available must increase to allow for multiple threads to carry out preliminary batch-processing of data before handing it over to TPU (so this unit doesn't have to wait to carry out these operations too). In this way, simplicity and control can be maintained over the framework.13 This would mean requiring access to more hardware which is costly.

Spark

Why Spark came about

Spark is a cluster-computing framework motivated in overcoming the limitations of the MapReduce cloud computing paradigm. Meanwhile retaining the scalability and fault tolerance of its inspired framework, Spark compensates for the linear data flow of MapReduce by incorporating iterative machine learning algorithms and adopting interactive data analysis tools. Spark specialises in solving problems that requires reusing datasets across parallel operations to allow iterative computation.14 The difference between the two is that Spark is able keep things in RAM where data is easily accessible, whereas MapReduce shuffles data in and out of disk memory.15 This in effect results in the programs running 100x faster than MapReduce in memory (or 10x faster on disk).16

Spark also introduces some new utilities, most importantly, integrated libraries to allow “machine learning, streaming, graph programming and SQL”.15 This is an added benefit compared to MapReduce as it allows the Spark framework to be more generalised than its restricted counterpart. The libraries increase the possible processing functions to work on the dataset and allows other explorative processing tasks into a working pipeline (as detailed in the next section) to work over the model to help tasks in relation to big data meanwhile increasing usability and performance.17

How the framework is utilised

As Spark incorporates new libraries, the framework can now be utilised in more real-life applications. Some of Spark’s new libraries include:16, 18

  • SQL and DataFrames: Allows you to query structured data in SQL or a DataFrame API .
  • MLlib: Machine Learning library allows Spark to batch process datasets offline with a given set of machine learning algorithms you can choose from.
  • GraphX: Contains graph-processing utilities to visualise data as graphs or collections.
  • Spark Streaming: Useful for complex real-time analytics from incoming data streams from multiple disparate sources. When combined with MLlib, you can then have a real-time machine learning system.

Since the libraries given allow interactive analysis and stores loaded data in RAM, users can load datasets and query them repeatedly on various virtual machines in parallel and still get multiple analyses completed in reasonable time rather than running separate jobs like MapReduce that involves reading and writing from the disk (which causes the low tolerance to latency issue in that framework).14

Spark can be used to simply calculate trends from data within a certain time frame or in real-time using Spark Streaming.18 This is useful for things like interactive and advanced analytics (used in Twitter’s real-time statistics) or high performance batch computation which can be utilised in cases such as Twitter’s real-time tweet analytics, data-mining in Netflix for recommendations or even for network security monitoring.15, 17

Limitations of Spark

Although it is much more of an efficient and generalised cloud computing framework than MapReduce, Spark still has drawbacks that make it ineffective for usage:19

  • No file management system: Spark does not have its own file manager so usually Spark will need to merge with Hadoop (Google’s open source MapReduce framework) Distributed File System (HDFS) and another cloud-based data platform in order to function.
  • No support for real-time processing: Spark Streaming is close to but is not real-time processing as the live data gets divided into smaller workloads that get processed and the results are returned in batches (micro-batch processing). The benefit of real-time processing is that every process is directly reflected in the master file of the data so it is always updated yet Spark does not support this entirely.
  • Cost-effectiveness: The operations of the framework that is performed on the data is done in memory so when working with Spark, there is high memory consumption so a large RAM unit is required. In comparison to MapReduce (in which disk space is cheaper), expenses for Spark is high.
  • High latency with data streams: There is a large delay in data delivery during computation and therefore less data gets through per unit time (lower throughput). In big data analysis, computations will get performed on data streams and this can hinder Spark's performance if the framework has a low tolerance to latency and results produced may not be so reliable.20
  • Back pressure handling: At a data input/output, there may be a build-up of data in memory that prevents any incoming data from being transferred until we can empty it which delays computation.
